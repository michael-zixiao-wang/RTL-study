# 异步复位同步释放

**“异步复位，同步释放”**几乎是处理全局复位信号的唯一工业标准。引入这个机制的原因是在触发器的底层物理特性中，复位引脚和数据引脚（D 端）一样，也有极其严格的时序要求。对于复位信号的撤销（Release），存在两个致命的时间窗口：
- 恢复时间 (Recovery Time)： 类似于建立时间 (Setup Time)。复位信号撤销的时刻，必须早于时钟上升沿到来前的一个特定时间段。
- 撤除时间 (Removal Time)： 类似于保持时间 (Hold Time)。复位信号撤销的时刻，必须晚于时钟上升沿之后的一个特定时间段。

1. 纯异步复位的问题

- 做法： always @(posedge clk or negedge rst_n)
- 优点： 复位瞬间生效，不需要等时钟沿。即使时钟彻底死掉，依然能把系统清零，极其安全。
- 致命缺点（释放时的亚稳态）： 当复位按键松开（撤销复位）时，这个动作是随机的。如果它恰好落在恢复时间或撤除时间的违例窗口内，触发器就会陷入亚稳态。更可怕的是，芯片内有上百万个触发器，由于时钟树偏斜和布线延迟，有的触发器可能在这个周期解除了复位，有的触发器可能在下个周期才解除复位。整个状态机瞬间四分五裂，系统直接死机。

2. 纯同步复位的困局

- 做法： always @(posedge clk)，把复位写在 if 里。
- 优点： 100% 同步电路，完全由时钟沿驱动，绝对不会产生恢复/撤除时间违例，时序分析极其干净。
- 致命缺点： 
    - 必须依赖时钟： 如果系统刚上电，锁相环（PLL）还没稳定，或者系统处于低功耗休眠模式（时钟被关断），这时候你按复位键，系统毫无反应。
    - 增加组合逻辑面积： 综合工具必须把同步复位信号作为数据输入（D 端）的一部分，在触发器前面插入额外的与门/多路选择器，导致数据通路的延迟增加。

为了同时拥有异步复位的**立即生效（不依赖时钟）特性，以及同步复位的绝对安全（不违例）**特性，前辈们发明了这个机制：
- 复位按下时： 狂暴模式，无视时钟，瞬间将所有触发器清零。
- 复位松开时： 绅士模式，不立刻起效，而是经过一个“复位同步器”，乖乖等到下一个时钟边沿到来时，与时钟对齐后再撤销复位。

# 复位信号产生电路（复位同步器）

```sv
module reset_synchronizer (
    input  wire clk,          // 系统目标时钟
    input  wire rst_async_n,  // 外部输入的原始异步复位信号 (低有效)
    output wire rst_sync_n    // 输出给内部全系统使用的“异步复位同步释放”信号
);

    // 定义两级同步寄存器
    reg rst_sync_reg1;
    reg rst_sync_reg2;

    // 核心逻辑：这里依然是异步复位的 always 块！
    always @(posedge clk or negedge rst_async_n) begin
        if (!rst_async_n) begin
            // 【异步复位阶段】
            // 当外部 rst_async_n 拉低时，不管有没有时钟，
            // 这两个寄存器瞬间被清零。
            // rst_sync_n 会立刻跟着变低，全系统瞬间复位。
            rst_sync_reg1 <= 1'b0;
            rst_sync_reg2 <= 1'b0;
        end else begin
            // 【同步释放阶段】
            // 当外部 rst_async_n 拉高（释放）时，不会立刻执行。
            // 必须等待 posedge clk 也就是时钟上升沿到来！
            // 第一个时钟沿：把高电平 1'b1 打入第一拍。
            // 第二个时钟沿：把高电平 1'b1 打入第二拍，此时 rst_sync_n 才真正变高。
            rst_sync_reg1 <= 1'b1;           // 第一级 D 端永远绑死在 1'b1 (高电平)
            rst_sync_reg2 <= rst_sync_reg1;  // 第二级接收第一级的状态
        end
    end

    // 最终输出同步化后的复位信号
    assign rst_sync_n = rst_sync_reg2;

endmodule
```

在你的整个芯片/FPGA 顶层中，正确的连线规范如下：
- 把物理管脚进来的复位信号，接入 reset_synchronizer 模块。
- 在业务逻辑（比如你的 ALU、FIFO、状态机）中，继续使用标准的异步复位代码风格，但是把信号换成同步化之后的 rst_sync_n：

```sv
// 业务模块代码风格绝对不要变！依然是纯异步复位写法
always @(posedge clk or negedge rst_sync_n) begin
    if (!rst_sync_n) begin
        // ... 清零
    end else begin
        // ... 正常逻辑
    end
end
```
